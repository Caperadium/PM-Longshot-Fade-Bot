# Changes

- Add a typed repository layer (`fader/persistence/repos.py`: `Db`, `PositionsRepo`, `OrdersRepo`, `BreakerRepo`, `DecisionsRepo`, `ControlRepo`, `EngineStateRepo`, `ConfigKVRepo`) and migrate all engine-side SQL behind it, removing duplicated raw-SQL call sites from `engine/risk.py`, `execution/idempotency.py`, `execution/order_manager.py`, `engine/reconciler.py`, `engine/strategy_loop.py`, `engine/state_publisher.py`, `engine/pollers.py`, `engine/control_consumer.py`, `persistence/decision_log.py`, `config/config_loader.py`, and the two flagged spots in `dashboard/app.py`. Order-fill bookkeeping (order status update + position insert) now runs inside one `Db.transaction()` so a crash mid-write can no longer leave a FILLED/PENDING order with no matching position row. `decision_log.log_decision` now returns `bool`; `order_manager` logs a warning when a decision row fails to persist. `engine.risk.get_open_notional` becomes a thin delegate to `PositionsRepo.deployed_total` (kept for `provider.py`/`strategy_loop.py`, repointed in a later phase). Adds `fader/tests/test_repos.py` (32 tests: every repo method, transaction commit/rollback, atomic fill-bookkeeping commit and rollback-on-failure).

- Phase 4 of the architecture refactor (shared filter core): new package `fader/strategy/` with `filters.py` -- pure, I/O-free implementation of entry filters 1-8 (NO ask in band, DTE, continuously-in-band, 24h volume, cumulative volume, book depth, staleness/gap, no doubling up), shared verbatim by the live engine and the backtest engine for the first time. `FilterParams` (frozen dataclass) carries thresholds plus three policy flags: `check_staleness` and `check_time_in_band` (both False in paper mode -- the two existing paper carve-outs, now expressed as explicit params instead of scattered `cfg.mode != "paper"` checks) and `missing_dte: "reject"|"skip"` (live rejects a missing DTE as `dte_out_of_range`; backtest skips the DTE filter entirely, preserving its known fail-open divergence pinned by `test_backtest_engine.py`). `EntrySnapshot` documents per-field None semantics as code, not just comments: `best_ask=None` always rejects `no_book`; `dte=None` follows `missing_dte`; `volume_24h`/`volume_total`/`ask_depth_usd`/`is_stale = None` are each independently recorded in `FilterResult.skipped` rather than rejecting or silently passing (no generic "None means pass" rule exists anywhere in the module). - Fix two pre-existing bugs surfaced during the architecture refactor (`temp/bugfix-plan.md`):
  (1) **Reaper TTL timestamp-format mismatch**: `OrdersRepo.reap_stale_unknown` compared `created_at` (written as `datetime.isoformat()`, e.g. `...T07:05:11.231837+00:00`) directly against SQLite's `datetime(?, '-3600 seconds')` output (`YYYY-MM-DD HH:MM:SS`, space-separated). Raw string collation put `'T' (0x54) > ' ' (0x20)`, so any same-UTC-date `created_at` compared GREATER than the cutoff regardless of time-of-day -- the reap only fired once the date prefix itself differed, making the effective TTL ~1 day instead of the intended 1 hour. Fixed by computing `cutoff_iso` in Python (`datetime.fromisoformat(now_iso) - timedelta(seconds=UNKNOWN_ORDER_TTL_S)`, new module-level constant, default 3600s) and comparing it as a string directly against `created_at` (`created_at < ?`) -- no SQLite date parsing involved, since a parser failure there returns NULL and silently makes the WHERE clause false (the same failure mode, better hidden). Signature of `reap_stale_unknown(now_iso, conn=None)` is unchanged. **Intentional behavior change**: the reaper now genuinely fires at ~1 hour instead of ~1 day. Since `CANCELLED` is a retryable status for idempotency-key purposes (`execution/idempotency.py`), this also means re-entry on the same token/params becomes possible as soon as 1 hour after an UNKNOWN order event, where previously it took up to a day -- if an UNKNOWN order was in fact still live on the exchange at that point, double exposure is possible. This is the originally-specified 1h TTL design; the earlier Phase 6 reap-commit fix had already accepted this risk at the (buggy) ~1-day granularity, this fix only tightens the window to what was always intended. New `TestReaperTtlPinned` class in `fader/tests/test_paper_resolution.py` (4 tests, ALL using a pinned synthetic `now_iso` rather than wall-clock, since a wall-clock test could spuriously pass pre-fix depending on time-of-day): 2h-before-same-date reaped (written first and demonstrated failing against pre-fix code), 30-minutes-before not reaped, 61-minutes-before boundary reaped, non-UNKNOWN statuses untouched.
  (2) **Reconcile escalation blind spot**: `Reconciler._reconcile_orders`' `fetch_open_orders() -> None` path (API error) logged a warning and returned with no counter, no `engine_state` signal, and no telegram alert -- a persistent order-API outage produced warnings forever with no escalation, unlike the existing positions/paper-resolutions escalation mechanism. Added a parallel `_order_reconcile_failures` counter (kept separate from the existing `_reconcile_failures`, since orders and positions/paper-resolutions are independent failure surfaces) with an `order_reconcile_failures` property; on `None`, increments the counter, publishes `order_reconcile_failures` to `engine_state`, and calls the existing `infra.telegram.alert_reconcile_failures(count, context, error)` with `context="reconcile_orders"` (no signature change to the alert function). Fire semantics match the existing mechanism exactly: alert fires on every cycle at or above the threshold (5), not once at the crossing. Resets to 0 (and re-publishes) on a fully completed reconcile-orders cycle; no new catch-all was added (the method still has no try/except around its body -- the counter only moves on the `None`-skip path). Paper mode returns before the fetch, so the counter never moves there. New `TestOrderReconcileFailureEscalation` class in `fader/tests/test_paper_resolution.py` (3 tests): `None` x6 -> counter climbs 1..6 and alert fires exactly twice (at 5 and at 6, pinning the fire-repeatedly `>=` semantics that a x5-only test cannot distinguish from fire-once); `None` x3 then a successful cycle -> counter resets to 0, no alert; paper mode -> counter stays 0.
  Files: `fader/persistence/repos.py`, `fader/engine/reconciler.py`, `fader/tests/test_paper_resolution.py`. Full suite: 380 passed (was 373; +7 new: 4 reaper + 3 escalation).
