You are an expert quant developer. Build a Polymarket long-shot fader bot and dashboard, deployable to a 24/7 VPS.

**Repo context.** This repo contains a separate project. Reuse only the Polymarket API client, market data fetching modules, and any general backtesting scaffolding that is strategy-agnostic. Discard everything else. Do not preserve unused code "just in case."

**Strategy.** Tested thesis: buy NO contracts priced between configurable bounds (default 0.80–0.95) and hold to resolution. No active exit logic — positions close on market resolution, at which point realized PnL is booked and the contract is marked closed. Polymarket API is the source of truth for fills, positions, bankroll, and resolution status.

**Market data.** Use Polymarket's CLOB websocket as primary data source. Maintain a persistent websocket connection subscribed to all enabled slugs, with in-memory order book state per contract. The strategy decision loop runs on a configurable interval (default 1s) and reads from in-memory state; it does not drive websocket polling. Polling is used only for: bankroll reconciliation (default 30s), market resolution status (default 60s), and new-contract discovery within multi-outcome markets (default 5 min). Implement websocket reconnection with exponential backoff and full book resync on reconnect. Per-contract staleness check: if no websocket update for that contract in `max_staleness_seconds` (default 30s), exclude from entry eligibility until fresh data arrives. On any data gap exceeding `gap_halt_seconds` (default 60s) across the feed, halt new entries until feed health is confirmed. Track `min_time_in_band` as a continuous interval over the websocket-observed price path, reset on any out-of-band tick.

**Entry logic.**

For each market in `slugs.csv`, for each NO contract:

1. Eligibility filters (all configurable, all must pass):
   - Price within band `[band_low, band_high]`, defaults 0.80–0.95.
   - DTE within `[min_dte, max_dte]`.
   - Price has been continuously within band for at least `min_time_in_band` (default 10 minutes), measured over websocket-observed price path.
   - Market 24h volume ≥ `min_24h_volume` (default $1,000).
   - Market cumulative volume ≥ `min_total_volume` (default $10,000).
   - Order book depth at touch ≥ `min_book_depth`.
   - Contract data freshness: last websocket update ≤ `max_staleness_seconds`.
   - No existing position on this exact contract (no doubling up).
   - Per-market exposure cap not breached (default 5% of bankroll per market, configurable).
   - Total deployed bankroll cap not breached (configurable).
   - Daily loss circuit breaker not tripped (configurable).

2. Order placement:
   - If spread ≤ 1c: market order at fixed dollar notional (configurable).
   - If spread > 1c: limit order at mid, fixed dollar notional. Requote when mid moves ≥ 0.5c. Cancel-and-replace with new idempotency key. TTL 5 minutes, then cancel and re-evaluate next cycle. Cancel immediately if price exits band.

3. Multiple contracts per market are permitted (subject to per-market cap). Different markets are independent.

**Exit logic.** None until resolution. On resolution, reconcile against Polymarket API, mark contracts closed, book realized PnL.

**Slugs database (`slugs.csv`).** Columns: `slug`, `enabled` (bool), `band_low` (optional override), `band_high` (optional override), `size_override` (optional), `added_at`, `notes`. **Infrastructure.**

- Polymarket credentials in `.env`, never logged, never sent to dashboard.
- State recovery on restart: reconcile against Polymarket API as ground truth. Idempotency keys on all order submissions so restart-driven retries don't double-fill. Websocket resubscribes to all enabled slugs on startup and rebuilds in-memory book state before strategy loop begins.
- Rate limiting: client-side token bucket below documented Polymarket REST limits. Separate queues for reads and writes, writes prioritized. Exponential backoff with jitter on 429.
- Heartbeat to Telegram every N minutes (configurable). Telegram alerts on: bot start/stop, circuit breaker trip, order rejection by exchange, position opened, position resolved, websocket disconnect/reconnect, gap halt triggered, unhandled exception.
